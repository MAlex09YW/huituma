# 信息查询页（三图批量・后台保护版）

同时准备三张成品图，每张拥有独立二维码、独立表单验证码、独立随机编号和独立 AES-GCM 解密钥匙。二维码由软件先生成，用户手动将它放入对应的最终图片。所有二维码的第一个验证页面共用一张全站固定顶部图片；验证成功后的第二个页面才解密并显示对应的带二维码成品图。

## 页面分工

- `https://www.henanshebaogov.com/admin/` 是发布后台，应使用 Cloudflare Access 只允许发布者本人登录。
- `https://www.henanshebaogov.com/?view=...&form=...#k=...` 是新二维码地址。扫码后，第一个页面显示全站固定顶部图片和验证码框；验证正确后才进入只显示对应成品图的第二个页面。
- 直接访问域名首页不会出现发布器，也没有任何介绍，只显示空白页。
- 旧二维码继续使用原来的根路径和加密格式，不需要因为后台迁移而重做。

## 更新 GitHub Pages 网站

将本项目的整个 `docs` 文件夹和 `README.md` 上传到仓库根目录并覆盖旧版本。保留 `docs/vault` 中已有的 `.bin` 文件。

GitHub Pages 发布源继续使用：

- Branch：`main`
- Folder：`/docs`

上传完成后，发布后台的新入口是 `/admin/`。在 Cloudflare Access 配置完成之前，这个入口暂时仍可被直接访问；应尽快按随包说明把 `www` 记录切换为 Cloudflare 代理，并只保护路径 `admin`。

## 三图制作流程

1. 先在后台的“固定验证页顶部图片”窗口选择一张图片，点击“上传并设为固定图片”。除非重新上传替换，否则所有二维码一直显示这一张。
2. 在图片 1、图片 2、图片 3 中分别点击“生成并下载”。每个二维码都会得到唯一绑定的 32 位表单验证码。
3. 三个二维码不可混用；分别手动放进对应的最终图片并导出成品图。
4. 在每个任务位选择对应的带二维码成品图。
5. 可以点击“加密并发布这张”，也可以点击顶部“发布全部已准备图片”。
6. 扫码后的第一个页面会自动填好表单验证码；输入图片验证码并点击橙色“查询”，正确后才请求并进入第二个页面显示最终图片。输入错误不会弹提示，而是清空输入并刷新验证码。

旧版单任务的未完成项目会自动迁移到新版的“图片 1”。已经发布的旧 `.bin` 文件和旧二维码继续兼容。

## 自定义域名

本项目的主域名已设为 `https://www.henanshebaogov.com/`，`docs/CNAME` 已包含 `www.henanshebaogov.com`。在 Cloudflare DNS 和 GitHub Pages 完成绑定、确认 HTTPS 正常后，再生成新的二维码。

域名一旦写入二维码，旧二维码不会自动改址。使用自己拥有的域名可以降低对某一个托管账号的依赖：如果原托管账号失效，可将完整备份恢复到其他静态服务器，再把同一域名指向新服务器，旧二维码仍可继续使用。

这里的“旧二维码仍可使用”只适用于二维码里已经写入自有域名的情况。当前写着 `malex09yw.github.io` 的二维码仍依赖该 GitHub 地址；重要图片应在自有域名生效后重新生成二维码并重新发布。

## 隐私

- 发布后台的真正访问限制由 Cloudflare Access 在服务器入口处执行，不能用前端密码代替。
- 图片在本机浏览器中加密，公开仓库或服务器只保存随机文件名的 `.bin`。
- 全站固定顶部图片需要在验证前显示，因此会以普通 PNG 存在网站中，不属于加密成品图；不要把需要保密的内容放进这张固定图片。
- 解密钥匙只在二维码链接的 `#` 后面，不会随网页请求发送给服务器。
- 二维码相当于查看钥匙；拿到二维码、二维码照片或完整扫码链接的人可以看图。
- 图片验证码用于扫码后的操作步骤确认，不是服务器端登录或强访问控制；持有完整二维码链接的人仍持有解密钥匙。
- GitHub token 只用于自动上传，绝不应放入备份、二维码或聊天记录。

## 完整备份应包含

- `docs` 中的网站程序文件。
- `docs/vault` 中所有已发布的 `.bin` 加密文件。
- 网站根目录中的 `verification-header.json` 和其指向的 `verification-header-*.png` 固定顶部图片。
- 已经制作和分发的二维码图片。
- 从网页顶部“导出任务备份”得到的 JSON；它包含二维码密钥，不包含 GitHub token，应离线私密保管。
- 域名注册商、DNS 配置和 GitHub Pages 自定义域名记录的文字备忘。

恢复站点不需要原 GitHub 或 Codex 账号，但要从互联网访问，仍需任意一个在线静态服务器。恢复时必须保留原域名、原 `.bin` 文件名和 `vault` 目录结构。
